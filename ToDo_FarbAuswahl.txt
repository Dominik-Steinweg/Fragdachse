"Du bist ein Senior Multiplayer Game Developer. Wir bauen ein Farb-Auswahl-System für unser Topdown-Shooter-Lobby-Overlay (welches in der gleichen Phaser-Szene wie die Arena läuft, setDepth ganz oben).

Es gibt exakt 12 vordefinierte, auswählbare Farben. Nutze dafür unsere Konstanten: BLUE_2, BLUE_5, GREEN_2, GREEN_5, BROWN_2, BROWN_5, GOLD_2, GOLD_5, RED_2, RED_5, PURPLE_2 und PURPLE_5.

Jeder Spieler MUSS eine einzigartige Farbe haben (keine Duplikate). Bitte implementiere dieses System mit strikter Host-Autorität über das Playroom Kit, um Race Conditions zu verhindern:

1. Der Room State (Host Only):

Beim Start initialisiert der Host (if (Playroom.isHost())) ein Array availableColors im Playroom Room-State, das exakt diese 12 Hex-Werte enthält.

Auto-Assign: Wenn ein neuer Spieler joint, schnappt sich der Host zufällig eine Farbe aus availableColors, entfernt sie aus dem Array, updatet den Room-State und speichert die Farbe im Player-State dieses Spielers (Key: customColor).

Disconnect: Verlässt ein Spieler das Spiel, muss der Host dessen customColor zwingend wieder zurück in das availableColors-Array pushen und den Room-State updaten.

2. Der Farbwechsel (RPC-Logik & Race-Condition-Schutz):

Der Client darf seine Farbe NICHT selbst im State ändern.

Wählt ein Spieler eine Farbe im UI, sendet er einen RPC requestColorChange mit der Wunschfarbe an den Host.

Der Host empfängt den RPC und prüft, ob die Farbe noch in availableColors ist.

Wenn JA: Host entfernt die neue Farbe aus dem Pool, legt die ALTE Farbe des Spielers in den Pool zurück, aktualisiert den Room-State und setzt die neue Farbe im Player-State. Sende RPC colorAccepted an den Client.

Wenn NEIN: Sende RPC colorDenied an den Client.

3. Das Reaktive Phaser UI (Lobby Overlay):

Baue das Popup in Phaser als Container (mit Hintergrund und klickbaren Farb-Rechtecken).

Echtzeit-Reaktivität: Das Popup muss sich sofort visuell updaten wenn sich die verfügbaren Farben ändern. Es dürfen IMMER nur die Farben als klickbare Buttons angezeigt (oder aktiviert) werden, die aktuell im Array availableColors stehen.

Feedback: Zeige die aktuelle Farbe des lokalen Spielers deutlich an. Empfängt der Client colorDenied, lass das UI kurz visuell warnen (z.B. Wackeln via Tween).

4. Projektile einfärben:

Passe die Projektil-Logik an. Wenn ein Spieler schießt, muss das Projektil zwingend die aktuelle customColor dieses Spielers übernehmen (z.B. via setFillStyle oder setTint).

Bitte zeige mir den Code für die Host-Logik (Initialisierung und RPC), das reaktive Phaser-UI und die Projektil-Anpassung."
